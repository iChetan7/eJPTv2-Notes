 ## Introduction to Web Applications

A foundational overview of **how websites work**, covering the core technologies (HTML/CSS/JS), HTTP protocol mechanics, headers, methods, sessions, cookies, and the critical difference between HTTP and HTTPS — all essential for web application penetration testing.

---

## What is a Website?

A website is **files hosted on a server** that a browser requests and renders:

| File Type | Purpose |
|---|---|
| **HTML** | Structure and content (HyperText Markup Language) |
| **CSS** | Presentation and styling (Cascading Style Sheet) |
| **JavaScript** | Functionality and interactivity |
| **PHP** | Backend server-side processing |

## HTTP Protocol Basics

### Request → Response Model
```text
Client (Browser)          Server
     |                         |
     |---- GET /login.php ---->|
     |                         |
     |<<--- 200 OK + HTML ------|
     |                         |
```

### Common HTTP Methods

| Method | Purpose | Pen Test Relevance |
|---|---|---|
| **GET** | Retrieve a file/resource | Most common; parameters in URL |
| **HEAD** | Retrieve headers only | Recon — check if resource exists without body |
| **POST** | Submit data (forms, logins) | Credentials, file uploads |
| **PUT** | Upload file to server | **Dangerous if enabled** — can upload shells |
| **DELETE** | Remove file from server | **Dangerous if enabled** |
| **OPTIONS** | List available methods | Recon — find enabled methods |
| **CONNECT** | Tunnel connection | Proxy abuse |
| **PATCH** | Partial resource update | API testing |
| **TRACK** | Echo back request | XST (Cross-Site Tracing) attacks |

## HTTP Headers

### Request Headers (Client → Server)
| Header | Purpose |
|---|---|
| **Host** | Target domain/IP |
| **User-Agent** | Browser/client identification — **can be spoofed** |
| **Cookie** | Session data sent with every request |

### Response Headers (Server → Client)
| Header | Purpose |
|---|---|
| **Status Code** | Result of the request |
| **Content-Type** | Type of data being returned (text/html, application/json) |
| **Location** | Redirect target (used with 302) |

---

## HTTP Status Codes

| Code | Meaning | Example |
|---|---|---|
| **200 OK** | Request successful | Page loaded |
| **302 Redirect** | Resource moved temporarily | Login success → redirect to dashboard |
| **404 Not Found** | Resource doesn't exist | Broken link, hidden dir not found |
| **401 Unauthorized** | Authentication required | Admin panel without login |
| **403 Forbidden** | Access denied | Directory listing disabled |
| **500 Internal Server Error** | Server malfunction | SQL injection payload broke query |

---

## Sessions & Cookies

HTTP is **stateless** — it doesn't remember you between requests. **Cookies** solve this:

| Cookie Type | Purpose |
|---|---|
| **Session Cookie** | Maintains login state (e.g., `PHPSESSID`) |
| **Authentication Cookie** | Proves who you are |
| **Tracking Cookie** | Analytics/advertising (privacy concern) |

**Pen Test Angle**: Steal someone's session cookie = impersonate them without credentials.

## HTTP vs HTTPS

| | HTTP | HTTPS |
|---|---|---|
| **Port** | 80 | 443 |
| **Encryption** | None | TLS/SSL |
| **Wireshark View** | Plaintext credentials | Gibberish (encrypted) |
| **Security** | Vulnerable to sniffing | Protected in transit |

### Wireshark Proof — HTTP Sniffing
```bash
# Start Wireshark on eth1
# Login to http://demo.ine.local with bee:bug

# Filter: frame contains "bee"
# Result: POST request with plaintext credentials visible
POST /login.php HTTP/1.1
login=bee&password=bug&security_level=0&form=submit
```

### HTTPS — Same Login, Encrypted
```bash
# https://demossl.ine.local
# Wireshark shows: TLS handshake → encrypted data
# No plaintext credentials visible
```

## Directory Busting with DIRB

```bash
# Basic directory enumeration
dirb http://demo.ine.local

# Default wordlist: /usr/share/dirb/wordlists/common.txt
```

### Common Findings

| Directory | What It Reveals |
|---|---|
| `/admin` | Admin panel — potential privilege escalation |
| `/images` | Static assets — info disclosure possible |
| `/javascript` | Client-side code — logic flaws, API endpoints |
| `/robots.txt` | **Goldmine** — lists directories the site wants hidden from search engines |

### robots.txt — Critical Recon
```text
# Example from demo.ine.local
User-agent: GoodBot
Disallow:                    # Nothing blocked

User-agent: BadBot
Disallow: /                  # Block everything

User-agent: *
Disallow: /admin/            # Hide admin
Disallow: /documents/        # Hide docs
Disallow: /images/           # Hide images
Disallow: /passwords/        # HIDE PASSWORDS ← PEN TESTER ALERT
```

> **Rule**: Whatever they want to hide from Google is exactly what you want to find.

## Lab Commands Reference

```bash
# Basic recon
nmap -sV demo.ine.local
# Result: Port 80 (HTTP), 3306 (MySQL)

# View page source
curl -v http://demo.ine.local
# Note: curl doesn't follow redirects by default (shows 302)

# Follow redirects
curl -L http://demo.ine.local

# Directory busting
dirb http://demo.ine.local

# Check robots.txt
curl http://demo.ine.local/robots.txt
```

## Key Takeaways

| Item | Detail |
|---|---|
| **Websites are just files** | HTML, CSS, JS, PHP served by a web server |
| **HTTP is stateless** | Cookies maintain sessions — steal cookie = steal session |
| **HTTP vs HTTPS** | Port 80 vs 443; plaintext vs encrypted; Wireshark proves it |
| **DIRB finds hidden dirs** | `/admin`, `/robots.txt`, `/passwords` — common goldmines |
| **robots.txt is recon** | "Disallow" entries reveal sensitive paths |
| **Methods matter** | OPTIONS shows what's enabled; PUT/DELETE = file upload/deletion risks |
