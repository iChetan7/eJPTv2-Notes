## What is this lab about?
A hands-on lab demonstrating how to enumerate a vulnerable web application, discover enabled HTTP methods, and interact with them using `curl`, `dirb`, and **Burp Suite Repeater**. The focus is on understanding what HTTP methods are exposed and how misconfigurations can lead to security risks.

---

## Key Concepts Explained

**HTTP Methods (Verbs)**
- **GET** — Retrieves a resource from the server (default browser request).
- **POST** — Submits data to the server (e.g., login forms).
- **HEAD** — Same as GET but returns only headers, no body.
- **OPTIONS** — Asks the server which methods are allowed on a resource.
- **PUT** — Uploads a file/resource to the server.
- **DELETE** — Removes a file/resource from the server.
- **TRACE / PROPFIND / LOCK / UNLOCK** — Less common; often used in WebDAV.

**Why methods matter:** If PUT/DELETE are enabled on directories that shouldn't allow it, attackers can upload webshells or delete critical files.

## Commands & Workflow

### Step 1 — Environment Setup
```bash
# Check your Kali IP address
ip addr show
# Example: 192.36.17.2 (attacker), target is 192.36.17.3
```

### Step 2 — Directory Enumeration with dirb
```bash
# Discover hidden directories and files
dirb http://192.36.17.3 /usr/share/dirb/wordlists/common.txt

# Key findings from scan:
# - /login.php      → Login form (POST)
# - /post.php       → Blog post page
# - /uploads/       → Writable directory (CRITICAL)
# - /.git/           → Source code exposure
# - /phpinfo.php    → PHP configuration leak
# - /server-status  → 403 Forbidden
```

### Step 3 — Manual HTTP Method Testing with curl

**GET request (fetch page)**
```bash
curl -X GET http://192.36.17.3/
# Returns: HTML content of the page
```

**HEAD request (headers only)**
```bash
curl -I http://192.36.17.3/
# Returns: HTTP/1.1 200 OK, Content-Type, PHP session cookie
```

**OPTIONS request (discover allowed methods)**
```bash
curl -X OPTIONS -v http://192.36.17.3/
# Returns: Allow: GET, HEAD, OPTIONS
# Note: PUT/DELETE NOT allowed on root
```

**OPTIONS on login.php**
```bash
curl -X OPTIONS -v http://192.36.17.3/login.php
# Returns: Allow: GET, POST, HEAD, OPTIONS
```

**OPTIONS on uploads directory**
```bash
curl -X OPTIONS -v http://192.36.17.3/uploads/
# Returns: Allow: GET, HEAD, POST, PUT, DELETE, TRACE, PROPFIND, PROPPATCH, LOCK, UNLOCK
# CRITICAL: PUT and DELETE are enabled!
```

**POST request (login simulation)**
```bash
curl -X POST -d "username=John&password=password" -v http://192.36.17.3/login.php
# Returns: HTTP/1.1 302 Found (redirect to index.php)
# Sets PHPSESSID cookie → successful login
```

**PUT request (file upload via curl)**
```bash
# First, create a test file
echo "Hello world" > hello.txt

# Upload it to the vulnerable /uploads/ directory
curl --upload-file hello.txt http://192.36.17.3/uploads/hello.txt
# Returns: HTTP/1.1 201 Created

# Verify upload
curl http://192.36.17.3/uploads/hello.txt
# Returns: Hello world
```

**DELETE request (file removal via curl)**
```bash
curl -X DELETE http://192.36.17.3/uploads/hello.txt
# Returns: HTTP/1.1 204 No Content
# File is now deleted
```

## Burp Suite Repeater Workflow

**Intercept and send to Repeater**
```bash
# 1. Set Firefox proxy to 127.0.0.1:8080
# 2. In Burp: Proxy → Intercept ON
# 3. Browse to target → request is caught
# 4. Right-click → Send to Repeater
```

**Testing methods in Repeater**
```http
# Change GET to OPTIONS in Repeater request line
OPTIONS / HTTP/1.1
Host: 192.36.17.3
# Response: Allow: GET, HEAD, OPTIONS

# Change to POST on root (should fail)
POST / HTTP/1.1
# Response: HTTP/1.1 405 Method Not Allowed

# PUT via Repeater (manual crafting)
PUT /uploads/hello.txt HTTP/1.1
Host: 192.36.17.3
Content-Length: 11

Hello world
# Response: HTTP/1.1 201 Created

# DELETE via Repeater
DELETE /uploads/hello.txt HTTP/1.1
Host: 192.36.17.3
# Response: HTTP/1.1 204 No Content
```

## HTTP Status Codes Reference

| Code | Meaning | Context in Lab |
|---|---|---|
| **200 OK** | Request successful | GET/HEAD/OPTIONS returned page |
| **201 Created** | Resource created successfully | PUT uploaded file |
| **204 No Content** | Success, no body to return | DELETE removed file |
| **301 Moved Permanently** | Resource redirected | `/uploads` → `/uploads/` (trailing slash) |
| **302 Found** | Temporary redirect | Login success → `index.php` |
| **404 Not Found** | Resource doesn't exist | Deleted file or wrong path |
| **405 Method Not Allowed** | HTTP method not permitted | PUT on root, POST on root |

## Key Takeaways

| Finding | Risk Level | Notes |
|---|---|---|
| `/uploads/` allows PUT/DELETE | **CRITICAL** | Can upload webshells or delete files |
| `/.git/` exposed | **HIGH** | Source code disclosure |
| `/phpinfo.php` accessible | **MEDIUM** | Information leakage |
| Login form uses POST | Normal | Standard authentication flow |
| Root only allows GET/HEAD/OPTIONS | **LOW** | Properly restricted |

- **Always check OPTIONS** on every discovered endpoint — methods vary by directory.
- **PUT enabled = file upload vulnerability** — potential for webshell deployment and RCE.
- **DELETE enabled = data destruction risk** — violates availability (CIA triad).
- **Burp Repeater** is superior to curl for rapid method testing and response analysis.
- Document all findings with **business impact**: What could an attacker do? (overwrite files, deface site, gain shell access).
