## What is this lab about?
A hands-on lab demonstrating **OWASP ZAP (ZAProxy)** for automated web application scanning, spidering, and vulnerability detection. Covers HUD setup, authenticated spidering, active scanning, and exploiting discovered vulnerabilities (XSS and SQL Injection) on the bWAPP vulnerable application.

---

## Key Concepts Explained

**OWASP ZAP (Zed Attack Proxy)**
- Free, open-source web app scanner maintained by OWASP.
- Combines proxy, spider, active scanner, and fuzzer in one tool.
- **HUD (Heads-Up Display)** — Overlay interface in the browser for real-time scanning and attack controls.

**Spider vs Active Scan**
- **Spider** — Crawls the site to discover all URLs, forms, and parameters (passive mapping).
- **Active Scan** — Sends malicious payloads to discovered endpoints to find vulnerabilities (can be noisy/dangerous).

**Context & Authentication in ZAP**
- **Context** — Defines the scope of what ZAP will scan (target domain/path).
- **Form-based Authentication** — ZAP can maintain session state by auto-logging in when sessions expire.
- **Forced User Mode** — Ensures ZAP stays authenticated while scanning.

## Commands & Workflow

### Step 1 — Target Discovery
```bash
# Identify target IP
nmap -sn 192.36.17.0/24
# Result: 192.36.17.3 (target), 192.36.17.2 (Kali)

# Verify open ports
nmap -sV 192.36.17.3
# Result: Port 80 (HTTP), Port 3306 (MySQL)
```

### Step 2 — Launch ZAP with HUD
```bash
# Start ZAP from terminal
zaproxy

# In ZAP:
# 1. Quick Start → Manual Explore
# 2. Enter URL: http://192.36.17.3
# 3. Enable HUD checkbox
# 4. Launch Browser (Chrome/Firefox)
# 5. Browser opens with ZAP HUD overlay
```

### Step 3 — Manual Navigation & Site Mapping
```bash
# Navigate through bWAPP application:
# - Login with credentials: bee / bug
# - Browse HTML Injection (Reflected GET)
# - Browse HTML Injection (Reflected POST)
# - Browse HTML Injection (Stored - Blog)
# - Browse SQL Injection (Search)

# ZAP automatically captures all requests in the Sites tree
```

### Step 4 — Configure Authentication Context
```bash
# In ZAP main window:
# 1. Find POST login.php in History/Sites tree
# 2. Right-click → Include in Context → Default Context

# 3. Go to Context → Default Context → Authentication:
#    - Method: Form-based Authentication
#    - Login Form Target URL: select the captured POST login.php
#    - Username Parameter: login
#    - Password Parameter: password
#    - Logged out indicator regex: Login (with capital L)
#      # This tells ZAP when session expires

# 4. Go to Users:
#    - Add user: bee / bug

# 5. Enable Forced User Mode:
#    - Click the user icon in top toolbar
#    - Select "bee" user
#    - Enable Forced User Mode (green indicator)
```

### Step 5 — Spider Scan (Authenticated)
```bash
# In ZAP:
# 1. Right-click target site → Attack → Spider
# 2. Select Context: Default Context
# 3. Select User: bee
# 4. Enable Recursive
# 5. Start Scan

# Result: Discovered 201 URLs across the application
# All authenticated pages now mapped in Sites tree
```

### Step 6 — Active Scan
```bash
# In ZAP:
# 1. Right-click target site → Attack → Active Scan
# 2. Select User: bee
# 3. Leave defaults, Start Scan

# Scan completes with findings:
# - 3 High (Critical)
# - 3 Medium
# - 3 Low
# - Multiple Informational
```

## Vulnerability Findings & Exploitation

### Finding 1: Persistent XSS (High)
```bash
# Location: HTML Injection - Stored (Blog)
# ZAP provides proof-of-concept payload

# Exploit:
# 1. Navigate to the vulnerable page via ZAP alert
# 2. In the blog entry field, inject:
<script>alert(1);</script>

# 3. Submit → Alert box pops up with "1"
# 4. Payload is stored and executes for all users viewing the blog
```

### Finding 2: Reflected XSS (High)
```bash
# Location: HTML Injection - Reflected (GET)
# ZAP provides replay URL with payload

# Exploit:
# 1. Click "Replay in Browser" from ZAP alert
# 2. Payload reflected in URL parameter executes in browser
# 3. Alert box confirms vulnerability
```

### Finding 3: SQL Injection (High)
```bash
# Location: SQL Injection (Search)
# ZAP payload: ZAP' OR '1'='1' --

# Manual Exploit:
# 1. Navigate to SQL Injection search page
# 2. In search field, enter:
ZAP' OR 1=1 --

# 3. Result: Returns ALL records from database
#    # The OR 1=1 makes the WHERE clause always TRUE
#    # Comment (--) ignores rest of original query

# URL-encoded version:
# ZAP%27%20OR%201%3D1%20--
```

## ZAP vs Other Tools Comparison

| Feature | OWASP ZAP | Burp Suite CE | Nessus |
|---|---|---|---|
| **Cost** | Free | Free (limited) | Paid |
| **Proxy** | Yes | Yes | No |
| **Spider** | Yes | Limited (CE) | No |
| **Active Scan** | Yes | No (CE) | Yes |
| **HUD** | Yes | No | No |
| **Authentication** | Form-based, script | Manual | Limited |
| **Best For** | Beginners, full workflow | Manual testing | Network scanning |

## Key Takeaways

| Finding | Severity | Details |
|---|---|---|
| Persistent XSS (Stored) | **High** | Blog comment field — payload stored server-side |
| Reflected XSS (GET) | **High** | URL parameter — payload reflected immediately |
| SQL Injection | **High** | Search field — `OR 1=1` dumps entire database |
| X-Frame-Options missing | Low | Clickjacking potential |
| Various info leaks | Info | Server versions, headers |

- **Always scope your scans** — Active scanning can crash applications or trigger alerts.
- **Authentication context is critical** — Without it, ZAP misses authenticated-only pages.
- **Forced User Mode** keeps ZAP logged in throughout long scans.
- **ZAP HUD** is beginner-friendly but the main interface offers more control.
- **Proof of Concept (PoC)** — ZAP provides replayable URLs; always verify findings manually.
- **SQL Injection impact**: Can expose usernames, passwords, customer data, financial records — complete database compromise.
- **Report integration**: Export ZAP findings and map to business impact (CIA triad) for client reports.
